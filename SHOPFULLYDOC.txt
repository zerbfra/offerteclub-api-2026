Webservice Offerista Group GmbH
This documentation describes the RESTful webservice of Offerista Group GmbH.

The base-url of the web-service is: https://delivery.offerista.com

Changelog
The changelog is available here.
Authentication
The authentication is based on the HTTP-basic authentication. You have to take the key as username and the secret as password. Due the encryption per SSL the informations are transferred securely.

GET-Requests
For all receivable resources two kinds of requests are supported. You can request a single-resource (example: https://delivery.offerista.com/companies/1) and a list-resource (example: https://delivery.offerista.com/companies?parameter1=value1&parameter2=value2), that includes a list of single resources. A single-resource-request contain one path-parameter - the required id - and several query-parameter, if the endpoint supports query-parameter for this kind of request. A list-resource-request contain several query-parameter to qualify the list-entries or to change other properties of the list-entries, f.e. the order of the entries. Query-parameter to qualify the list-entries are concatenate by an logical AND.

Some query-parameter could be specify multiple times within a single request. This array-query-parameter must be specified like https://delivery.offerista.com/companies?id=1&id=2&id=30. The parts of one array-query-parameter are concatenate by an logical OR.

The web-service support both JSON and XML as output format. To specify the required output format you have to set the "Accept"-header (request). (example: "Accept: application/xml") The type of the output within the response is described by the "Content-Type"-header (response).

Furthermore the web-service support GZIP-compression of the output. To enable the compression you have to set the "Accept-Encoding"-header (request). (example: "Accept-Encoding: gzip") The type of compression of the output within the response is described by the "Content-Encoding"-header (response).

Caching
A response from the web-service contains the HTTP-cache-header "Cache-Control: public" and "Cache-Control: max-age=xyz" and several application-specific "Vary"-header to support HTTP-caching.

To optimize the caching hit rate the following points should be implemented:

round the values of latitude and longitude to 4 decimal places
order the parameters of the requests alphabetically
PUT- and POST-Requests
PUT- and POST-requests create and modify single or multiple resources. The web-service support both JSON and XML as input format. You have to specify which input format you would take. To do so, you have to set the "Content-Type"-header (request). (example "Content-Type: application/xml")

Images
The web-service only delivers meta-information over an available image. You have to load this image by yourself from the images-management-system (IMS). The IMS is accessible via http and https. The url is build by the following parts:

http|https://{url}/[seo-text:]{id}_<{width}*n>x<{height}*n>[_fill<RRGGBB>][_mono][_nd].{type}
pattern	meaning
{value}	meta-information delivered by the web-service
value1|value2	alternatives
[value]	optional
<value>	variable value
valuen	multiple of value
Scaling:
To scale an image:
Calculate the scale factor based on the original height and width. Both dimensions you will receive from the api response. (see below)
Multiply the desired dimension (height or width) with the scale Factor to get the other dimension. (tolerance range is plus/minus one pixel)
The IMS-request will end up in a BadRequest, if the relation between height and width is not in the tolerance range and there is no "_fill" parameter.
Image Format
The provided image type (e.g., .webp in the example below) is just an example. WebP format is provided by default, but you are free to request images in any of the following formats, based on your needs:

webp
jpg
png
Specify your desired format by changing the file extension in the URL (e.g., .webp, .jpg, .png).

_nd Flag
You can use the _nd flag in the URL to enhance image quality:

For WebP images, the _nd flag will enable lossless compression.
For JPEG images, the _nd flag will increase quality by applying a lower compression rate.
Using the metadata from the example below, an IMS URL with the _nd flag enabled could look like this:

https://media.marktjagd.com/1_425x255_fillFFFFFF_nd.webp
Example:
You can create this valid IMS-url, if you deliver the following image-meta-information from the web-service :

https://media.marktjagd.com/1_425x255_fillFFFFFF.webp
<image>
  <url>media.marktjagd.com</url>
  <id>1</id>
  <width>500</width>
  <height>300</height>
  <type>webp</type>
</image>
Intelligent Radius
When you request industries, companies, stores and offers you should use our intelligent radius by omitting the explicit radius. With the intelligent radius you can support use cases like: The user finds offers of furniture stores within a radius of 50 km and offers of discounter within a radius of 10 km. The radius is specific for each industry or company.

Explicit radius: /stores?geo=51.9542,13.1234,100
Intelligent radius: /stores?geo=51.9542,13.1234

Language
To specify the desired content language use the "Accept-Language" header. This header should contain a language-tag that is conform to RFC 1766.

Resolve IP-address
To resolve an IP-address to a geo-coordinate you should use the resource location. The resource supports requests via POST and GET. If you use the POST method, you send the IP-address "application/x-www-form-urlencoded" encoded within the body, e.g. "ip=192.168.0.1". If you use the GET method, the client IP-address will be used to get the right location. If the process was successful you receive a location object with latitude and longitude (status code: 200 OK). Otherwise the status code 404 NotFound will be returned.

Feature Toggles
Features that are controlled by clients can be enabled by using the HTTP-header Active-Features.

Trackingpixel
The trackingpixel of the customer must be implemented in addition to the Offerista tracking. The customer checks the Offerista tracking by the number of trackingpixel requests. To implement the trackingpixel correctly the following points are important:

Currently the trackingpixel don't show to which Offerista tracking action they belong to. So at this time the trackingpixel must be called at the same time as the click tracking is sent.
Every object can have multiple trackingpixel. All trackingpixel must be called.
The integration of the trackingpixel depends on the kind of the integration:
A web integration must integrate the trackingpixel within a image tag.
An app integration must call the URL of the trackingpixel by a HTTP-GET request.
Trackingpixel have to be called by the client. A server side call would result into multiple trackingpixel request from the same ip-address or from a little range of ip-addresses.
Trackingpixel include the variable %%CACHEBUSTER%%. The variable has to be replaced by a unique value related to the time. For example the unixtimestamp. To avoid request failures the value of the variable should not include special signs. The cachebuster should avoid a client side caching.
Within an app integration the HTTP header User-Agent must be set correctly. To circumvent filtering by certain blacklists like the "IAB spiders & bots list" used by some customers (https://www.iab.com/guidelines/iab-abc-international-spiders-bots-list/), the header value must fulfill the following criteria:
it must contain the particular clients name and version (e.g. marktjagd/9.1)
it must not contain the name of HTTP clients like e.g. okhttp
The folowing example shows how the tracking pixel looks like: "http://trackingpixel.de/example.gif?cache=%%CACHEBUSTER%%"
Impression Trackers
Customer-provided tracking URLs can be included via the impression_trackers field. These trackers must be triggered in addition to the existing Offerista tracking and are used by the customer to validate impressions through their own systems.

Note on Similarities and Differences with trackingpixel
Similarities:

Both trackingpixel and impression_trackers are used for customer-side validation of user actions (e.g., views or clicks).
They require client-side triggering, either via image tag in HTML or HTTP-GET request, depending on whether it's a web or app integration.
They always include the placeholder %%CACHEBUSTER%%, which must be replaced on the client-side with a unique, timestamp-like value to prevent caching.
They support multiple URLs, all of which must be called.
For both fields, if no values are present, the field will be omitted entirely from the API response (i.e., the field will not appear with an empty list).
Differences:

trackingpixel is typically associated with click tracking and is called at the moment of the click, whereas impression_trackers are designed for view-based tracking and are called when the brochure or content is displayed.
trackingpixel can be defined on all offer types (products and brochures), while impression_trackers can only be defined on brochure-type offers.
Example impression tracker URL: https://analytics.example.org/pixel.gif?ref=brochure123&cache=%%CACHEBUSTER%%

/offers/brochures/redirect
GET
An usecase specific endpoint to receive a brochure for the redirect usecase.

The desired brochure is determined by the following fallback scenarios:

   0. If a brochureId was sent and the brochure exists it will be returned, independent from the other parameters.
   1. If a companyId was sent and case 0 did not returned a brochure and a brochure of the company exists
      (this also depends on the geo parameter) this brochure will be returned. In the case more than one brochure
      exists, the first depending on sort by "popularity, desc" will be taken.
   2. If a companyId was sent and case 0 and case 1 did not returned a brochure and a brochure of the industry of
      the company exists (this also depends on the geo parameter) this brochure will be returned. In the case more
      than one brochure exists, the first depending on sort by "popularity, desc" will be taken.
   3. If no brochure was returned by the previous cases and at least brochure exists (this also depends on the geo
      parameter) this brochure will be returned. In the case more than one brochure exists, the first depending
      on sort by "popularity, desc" will be taken.
   
If you send a geo parameter and add the "*" as radius the desired brochure three additional fallback scenarios are executed:

   1.1 If a companyId was sent and case 0 and case 1 did not returned a brochure and a brochure of the company
       exists, but not within the visibility-radius of the user this brochure will be returned. In the case more
       than one brochure exists, the first depending on sort by "popularity, desc" will be
       taken.
   2.1 If a companyId was sent and case 0, 1, 1.1 and 2 did not returned a brochure and a brochure of the
       industry of the company exists, but not within the visibility-radius of the user this brochure will be
       returned. In the case more than one brochure exists, the first depending on sort by "popularity, desc" will be taken.
   3.1 If no brochure was returned by the previous cases and at least brochure exists, but not within the
       visibility-radius of the user this brochure will be returned. In the case more than one brochure exists,
       the first depending on sort by "popularity, desc" will be taken.
